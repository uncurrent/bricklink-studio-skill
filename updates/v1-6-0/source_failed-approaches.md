# Подходы которые не сработали — и почему

Документ фиксирует конкретные технические решения, которые были попробованы в ходе разработки генератора Pocket-моделей, оказались неверными или недостаточными, и что именно из каждого в итоге пригодилось.

---

## 1. Редактирование model.lxfml вместо model.ldr

**Что делали:** Стали редактировать XML-файл `model.lxfml` внутри `.io`-архива, думая что именно он управляет 3D-моделью.

**Почему не сработало:** Stud.io рендерит `model.ldr` (LDraw-формат), а не LXFML. LXFML — вторичный метаданных-файл. Модель в окне просмотра не менялась вообще, хотя XML был корректно изменён.

**Что сработало:** Как только переключились на редактирование `model.ldr` — изменения сразу отобразились в Stud.io.

**Вывод:** Всегда работать только с `model.ldr`. LXFML можно игнорировать при программной генерации.

---

## 2. Y-axis only rotation (Pocket 4)

**Что делали:** При генерации задавали деталям только вращение вокруг Y-оси (`rot_y(angle)`). Предполагали, что детали будут выглядеть как случайно разбросанная куча.

**Почему не сработало:** Y — вертикальная ось. Вращение только вокруг неё означает, что все детали лежат горизонтально лицом вверх — как аккуратно разложенные, а не как рассыпанные. Ни одна деталь не стоит на боку, ни одна не перевёрнута.

**Что сработало:** Логика генерации позиций (Gaussian XZ + uniform Y), цветовая палитра, рецепт состава деталей — всё это перешло в следующие версии без изменений.

**Вывод:** Для реалистичной кучки необходимы наклоны по всем трём осям (Rx, Rz). Без них куча выглядит как витрина.

---

## 3. Фиксированные радиусы эллипсоида для collision detection (Pocket 5)

**Что делали:** Каждой детали назначили два фиксированных радиуса — `r_horizontal` (горизонтальный габарит) и `r_vertical` (вертикальный, примерно половина высоты детали). Плейту: `r_v = 5 LDU`. Кирпичу: `r_v = 14 LDU`. Проверяли пересечение: `dist_xz < rh1+rh2 AND |y1-y2| < rv1+rv2`.

**Почему не сработало:** Радиусы назначались исходя из *исходной* ориентации детали (лежит плоско). Но если плейт стоит на торце — его реальный вертикальный размер становится ~40 LDU, а не 5. Алгоритм этого не знал и считал что два торчащих вертикально плейта не пересекаются, хотя они физически насквозь друг через друга. Итог: ~25% деталей всё равно overlap.

**Что сработало:** Полноценная 3D-матрица вращения (R = Ry × Rx × Rz) — сама по себе правильная и перешла в v3 без изменений. Логика `tilt_strength` по категориям деталей — тоже осталась. Схема rejection sampling (пробуем N позиций, берём лучшую) — осталась.

**Вывод:** Bounding volume обязан учитывать текущую ориентацию объекта. Для произвольно повёрнутого прямоугольника правильный подход — rotation-aware AABB.

## 4. Uniform Y-distribution (Pocket 4–5)

**Что делали:** Y-координату (высота) брали из `random.uniform(Y_MIN, Y_MAX)` — равномерно по всей высоте стека.

**Почему не сработало полностью:** Детали получались равномерно "размазаны" по высоте. В реальной куче большинство деталей лежит внизу — под собственным весом. Верхние слои разреженные. Равномерное распределение делало кучу визуально "рыхлой" и неправдоподобной при взгляде сбоку.

**Что сработало:** Заменили на `triangular(Y_MIN, Y_MAX, Y_MIN + 40)` — треугольное распределение с пиком у дна. Большинство деталей теперь укладывается в нижнюю треть, единичные улетают выше.

**Вывод:** Треугольное или экспоненциальное распределение по Y значительно реалистичнее равномерного для имитации гравитации.

---

## 5. Gaussian clustering без учёта размера детали (Pocket 4–5)

**Что делали:** Позиции всех деталей сэмплировали из одного Gaussian распределения, независимо от размера детали.

**Почему не сработало:** Большие детали (плейт 2×8 = 160 LDU) и маленькие (кирпич 1×1 = 20 LDU) имеют принципиально разные footprint. Когда крупная деталь первой занимала центр — за ней туда же пытались попасть десятки мелких. В центре образовывался "затор" принудительных размещений. Самый первый крупный плейт мог занять 30–40% всего центрального пространства.

**Что сработало (частично):** Сортировка деталей — крупные размещаются первыми, мелкие последними. Мелкие заполняют дырки которые крупные оставляют по краям. Это снизило количество forced с ~25% до ~8%.

**Что не сработало полностью:** Gaussian по-прежнему агрессивно кластеризует всё в центре. Более равномерные стратегии — Poisson disk sampling или stratified grid — были бы лучше но сложнее в реализации с rejection sampling поверх.

**Вывод:** Порядок размещения важен не меньше чем алгоритм collision. Крупные — первыми.

## 6. Создание ZIP напрямую в примонтированной папке

**Что делали:** Пытались запустить `zip` прямо в `/sessions/.../mnt/BrickitStudio/Pockets/`.

**Почему не сработало:** Примонтированные папки workspace доступны на запись через file tools (Write, Edit), но для нативных shell-команд (zip, cp) могут быть read-only. Ошибка: `EROFS: read-only file system`.

**Что сработало:** Создавать ZIP в `/tmp`, потом копировать через `cp` в workspace.

```bash
cd /tmp && zip -r /tmp/PocketN.io model.ldr .info errorPartList.err
cp /tmp/PocketN.io "/path/to/workspace/Pocket N.io"
```

**Вывод:** Для любых операций генерации файлов — всегда использовать `/tmp` как рабочую область, workspace только для финального результата.

## 7. print() в stdout при генерации LDR

**Что делали:** В скрипте генерации использовали `print()` для вывода статистики (количество деталей, процент overlap и т.д.), одновременно записывая LDR в stdout через `sys.stdout.write()`.

**Почему не сработало:** Строки статистики (`// --- DONE: 71 parts ---`) попадали прямо в LDR-файл. Stud.io не смог корректно разобрать файл.

**Что сработало:** Redirect статистики в stderr: `print(..., file=sys.stderr)`. LDR писать напрямую в файл через `open(path, 'w')`.

**Вывод:** В генераторах файлов — никогда не смешивать диагностический вывод и данные в одном потоке.
